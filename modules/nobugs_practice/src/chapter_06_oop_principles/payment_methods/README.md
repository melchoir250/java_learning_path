# Задача 19: Платёжные методы (Интерфейс + Расширение)

## Описание проекта

Разработайте систему платёжных методов, демонстрирующую принцип расширяемости через интерфейсы. Система должна позволять
добавлять новые способы оплаты без изменения существующего кода клиентов. Все платёжные методы реализуют единый
интерфейс, что обеспечивает гибкость и слабую связанность.

## Структура проекта

```
payment_methods/
├── PaymentMethod.java      # Интерфейс платёжного метода
├── CreditCard.java         # Реализация оплаты кредитной картой
├── PayPal.java             # Реализация оплаты через PayPal
├── GooglePay.java          # Реализация оплаты через Google Pay
├── CryptoWallet.java       # Реализация оплаты криптовалютой (добавляется позже)
├── PaymentProcessor.java   # Класс для обработки платежей
├── Main.java               # Демонстрация работы
└── README.md               # Описание задачи
```

## Функциональные требования

### 1. Интерфейс `PaymentMethod`

Определяет контракт для всех платёжных методов:

- **Метод**: `void pay(double amount)` - выполняет оплату на указанную сумму
- **Метод**: `String getMethodName()` - возвращает название метода оплаты (опционально)
- **Метод**: `boolean validate(double amount)` - проверяет возможность оплаты (опционально)

### 2. Реализации платёжных методов

#### `CreditCard`

- Представляет оплату банковской картой
- **Поля**:
    - `String cardNumber` - номер карты
    - `String cardHolder` - владелец карты
    - `String expiryDate` - срок действия
    - `String cvv` - код безопасности
- **Конструктор**: для инициализации данных карты
- **Метод `pay()`**:
    - Проверяет валидность данных карты
    - Выполняет логику списания средств
    - Выводит информацию о транзакции

#### `PayPal`

- Представляет оплату через PayPal
- **Поля**:
    - `String email` - email аккаунта PayPal
    - `String password` - пароль (или токен)
- **Конструктор**: для инициализации учётных данных
- **Метод `pay()`**:
    - Проверяет авторизацию
    - Выполняет логику оплаты через PayPal API
    - Выводит информацию о транзакции

#### `GooglePay`

- Представляет оплату через Google Pay
- **Поля**:
    - `String deviceId` - идентификатор устройства
    - `String token` - токен авторизации
- **Конструктор**: для инициализации данных
- **Метод `pay()`**:
    - Проверяет наличие настроенного платёжного метода
    - Выполняет логику оплаты через Google Pay
    - Выводит информацию о транзакции

### 3. Класс `PaymentProcessor`

Отвечает за обработку платежей с использованием любого платёжного метода:

- **Метод `processPayment(PaymentMethod method, double amount)`**:
    - Принимает любой объект, реализующий `PaymentMethod`
    - Выполняет оплату через переданный метод
    - Обрабатывает возможные ошибки
- **Метод `processMultiplePayments(List<PaymentMethod> methods, double amount)`**:
    - Последовательно применяет несколько платёжных методов (опционально)

### 4. Класс `Main`

Демонстрирует работу системы:

- Создаёт экземпляры всех платёжных методов
- Создаёт платёжный процессор
- Выполняет оплату через разные методы
- Показывает, что код клиента не зависит от конкретной реализации
- Затем добавляет `CryptoWallet` и демонстрирует, что он работает без изменения существующего кода

## Технические требования

### Полиморфизм и расширяемость

- Все платёжные методы реализуют единый интерфейс
- `PaymentProcessor` работает с интерфейсом, а не с конкретными классами
- Новый платёжный метод можно добавить, не изменяя `PaymentProcessor` и существующий код

### Инкапсуляция

- Каждый платёжный метод инкапсулирует свою логику
- Детали реализации скрыты от клиентского кода

### Обработка ошибок

- Проверка корректности данных перед оплатой
- Обработка недостаточных средств
- Выброс исключения `PaymentException` при ошибке оплаты

### Пользовательское исключение (опционально)

- `PaymentException` - выбрасывается при ошибках оплаты
- Содержит информативное сообщение о причине ошибки

## Ключевые концепции

- **Полиморфизм через интерфейсы**: единый способ взаимодействия с разными платёжными методами
- **Расширяемость**: добавление новых методов без изменения существующего кода
- **Слабая связанность**: клиентский код зависит от интерфейса, а не от реализаций
- **Принцип открытости/закрытости (Open-Closed)**: открыт для расширения, закрыт для модификации

## Дополнительные требования (опционально)

1. **История транзакций**:
    - Добавить запись всех платежей в историю
    - Метод для получения истории платежей

2. **Комиссии**:
    - Разные платёжные методы могут иметь разную комиссию
    - Метод `calculateFee(double amount)` для расчёта комиссии

3. **Валюты**:
    - Поддержка разных валют
    - Конвертация валют

4. **Валидация**:
    - Более строгая проверка данных карты (формат, срок действия)
    - Проверка баланса перед оплатой

5. **Сравнение методов**:
    - Метод для сравнения платёжных методов по комиссии

6. **Поддержка QR-кодов**:
    - Добавить реализацию `QRPayment`

## Критерии оценки реализации

1. **Интерфейс**:
    - Чётко определённый контракт `PaymentMethod`
    - Все реализации корректно следуют контракту

2. **Расширяемость**:
    - Новый метод `CryptoWallet` добавляется без изменения существующих классов
    - `PaymentProcessor` не требует модификации

3. **Полиморфизм**:
    - Метод `processPayment` принимает параметр типа `PaymentMethod`
    - Один и тот же код работает с разными типами оплаты

4. **Инкапсуляция**:
    - Каждый класс инкапсулирует свою логику
    - Нет утечки деталей реализации

## Как должен работать код

### Этап 1: Базовая функциональность

1. Создаются объекты `CreditCard`, `PayPal`, `GooglePay` с соответствующими данными
2. Создаётся `PaymentProcessor`
3. Вызывается `processPayment()` для каждого метода оплаты
4. Каждый метод оплаты выполняет свою логику списания средств

### Этап 2: Расширение (без изменения существующего кода)

1. Создаётся новый класс `CryptoWallet`, реализующий `PaymentMethod`
2. Экземпляр `CryptoWallet` передаётся в `PaymentProcessor.processPayment()`
3. Код работает без каких-либо изменений в `PaymentProcessor` или других существующих классах
4. Система демонстрирует, что добавление нового функционала не требует изменения старого

## Принципы, демонстрируемые задачей

1. **Интерфейс как контракт**: единый способ взаимодействия для разных платёжных методов
2. **Полиморфизм**: возможность подставлять разные реализации в один и тот же код
3. **Расширяемость**: новая функциональность добавляется путём создания новых классов, а не изменения существующих
4. **Слабая связанность**: клиентский код зависит только от интерфейса
