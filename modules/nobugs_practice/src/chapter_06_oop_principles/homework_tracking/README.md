# Задача 34: Учёт домашек (Абстракция + Инкапсуляция)

## Описание проекта

Разработайте систему учёта домашних заданий, демонстрирующую принципы абстракции и инкапсуляции. Различные типы
заданий (Java, математика, дизайн) реализуют единый интерфейс `Task`. Преподаватель работает с заданиями через этот
интерфейс, не завися от конкретного типа задания. Это обеспечивает гибкость и возможность добавления новых типов заданий
без изменения существующего кода.

## Структура проекта

```
homework_tracking/
├── Task.java                 # Интерфейс домашнего задания
├── JavaTask.java             # Задание по Java
├── MathTask.java             # Задание по математике
├── DesignTask.java           # Задание по дизайну
├── Teacher.java              # Преподаватель (работает через интерфейс)
├── Main.java                 # Демонстрация работы
└── README.md                 # Описание задачи
```

## Функциональные требования

### 1. Интерфейс `Task`

Определяет контракт для всех домашних заданий:

- **Метод `void submit(String studentName, String solution)`** - сдать задание
- **Метод `void grade(int score, String feedback)`** - оценить задание
- **Метод `String getDescription()`** - получить описание задания
- **Метод `String getStatus()`** - получить текущий статус задания
- **Метод `String getStudentName()`** - получить имя студента (опционально)
- **Метод `int getScore()`** - получить оценку (опционально)

### 2. Реализации интерфейса

#### `JavaTask` (Задание по Java)

- **Поля**:
    - `String taskId` - идентификатор задания
    - `String description` - описание задачи
    - `String studentName` - имя студента
    - `String solution` - решение (код)
    - `int score` - оценка
    - `String feedback` - отзыв преподавателя
    - `TaskStatus status` - статус задания (NEW, SUBMITTED, GRADED)
    - `String[] testResults` - результаты тестов (опционально)
- **Особенности**:
    - Проверка решения может включать компиляцию кода
    - Может содержать ожидаемые тесты

#### `MathTask` (Задание по математике)

- **Поля**:
    - `String taskId` - идентификатор задания
    - `String description` - описание задачи
    - `String studentName` - имя студента
    - `String solution` - решение (формулы, вычисления)
    - `int score` - оценка
    - `String feedback` - отзыв преподавателя
    - `TaskStatus status` - статус задания
    - `String correctAnswer` - правильный ответ (опционально)
- **Особенности**:
    - Может проверять точность вычислений
    - Поддержка формул

#### `DesignTask` (Задание по дизайну)

- **Поля**:
    - `String taskId` - идентификатор задания
    - `String description` - описание задачи
    - `String studentName` - имя студента
    - `String solution` - ссылка на файл или описание
    - `int score` - оценка
    - `String feedback` - отзыв преподавателя
    - `TaskStatus status` - статус задания
    - `String filePath` - путь к файлу с работой (опционально)
- **Особенности**:
    - Поддержка прикрепления файлов
    - Оценка по критериям (композиция, цвет, техника)

### 3. Перечисление `TaskStatus` (Вспомогательное)

- **Значения**:
    - `NEW` - задание создано, не сдано
    - `SUBMITTED` - задание сдано, ожидает проверки
    - `GRADED` - задание проверено и оценено

### 4. Класс `Teacher`

Представляет преподавателя, работающего с заданиями через интерфейс:

- **Поля**:
    - `String name` - имя преподавателя
    - `List<Task> assignments` - список заданий (опционально)
- **Методы**:
    - `void reviewTask(Task task)` - проверить задание (запросить описание, статус)
    - `void gradeTask(Task task, int score, String feedback)` - оценить задание
    - `void assignTask(Task task, String studentName)` - назначить задание студенту (опционально)
    - `void printTaskInfo(Task task)` - вывести информацию о задании

### 5. Класс `Main`

Демонстрирует работу системы:

- Создаёт различные типы заданий (Java, математика, дизайн)
- Создаёт преподавателя
- Студенты сдают задания
- Преподаватель оценивает задания через единый интерфейс
- Показывает, что преподаватель работает с любым типом задания одинаково

## Технические требования

### Абстракция через интерфейс

- Все задания реализуют интерфейс `Task`
- `Teacher` работает с типом `Task`, а не с конкретными классами

### Инкапсуляция

- Каждый тип задания инкапсулирует свою специфическую логику
- Состояние задания защищено и изменяется только через методы

### Полиморфизм

- Методы `submit()` и `grade()` работают по-разному для разных типов заданий
- Преподаватель не знает о внутренней структуре задания

### Управление состоянием

- Статус задания изменяется последовательно: NEW → SUBMITTED → GRADED
- Нельзя оценить несданное задание

## Ключевые концепции

- **Абстракция**: выделение общего поведения в интерфейс
- **Полиморфизм**: единый интерфейс для разных типов заданий
- **Инкапсуляция**: сокрытие деталей реализации каждого типа задания
- **Слабая связанность**: преподаватель зависит только от интерфейса
