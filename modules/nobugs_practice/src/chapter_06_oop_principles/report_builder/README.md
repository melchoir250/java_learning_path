# Задача 26: Конструктор отчётов (Интерфейс + Композиция)

## Описание проекта

Разработайте систему для гибкого построения отчётов, демонстрирующую принципы композиции и расширяемости. Отчёт
собирается из независимых частей, каждая из которых реализует единый интерфейс. Конструктор отчётов объединяет эти
части, позволяя создавать различные структуры отчётов без изменения существующего кода.

## Структура проекта

```
report_builder/
├── ReportPart.java           # Интерфейс части отчёта
├── Header.java               # Заголовок отчёта
├── Table.java                # Таблица с данными
├── Footer.java               # Подвал отчёта
├── ReportBuilder.java        # Конструктор отчёта
├── Main.java                 # Демонстрация работы
└── README.md                 # Описание задачи
```

## Функциональные требования

### 1. Интерфейс `ReportPart`

Определяет контракт для всех частей отчёта:

- **Метод `String render()`** - возвращает строковое представление части отчёта
- **Метод `String getTitle()`** - возвращает название части (опционально)
- **Метод `int getOrder()`** - возвращает порядок отображения (опционально)

### 2. Реализации частей отчёта

#### `Header` (Заголовок)

- **Поля**:
    - `String title` - заголовок отчёта
    - `String subtitle` - подзаголовок (опционально)
    - `String date` - дата создания (опционально)
    - `String author` - автор отчёта (опционально)
- **Конструктор**: принимает заголовок и опционально другие параметры
- **Метод `render()`**:
    - Возвращает форматированный заголовок
    - Может включать линии разделители, рамки
    - Формат: `=== Заголовок ===`, дата, автор

#### `Table` (Таблица)

- **Поля**:
    - `List<String> headers` - заголовки столбцов
    - `List<List<String>> rows` - данные таблицы (строки)
    - `String caption` - подпись к таблице (опционально)
- **Конструктор**: принимает заголовки столбцов и опционально данные
- **Методы**:
    - `void addRow(List<String> row)` - добавить строку данных
    - `void setCaption(String caption)` - установить подпись
- **Метод `render()`**:
    - Возвращает отформатированную таблицу
    - Выравнивание столбцов по ширине
    - Границы и разделители

#### `Footer` (Подвал)

- **Поля**:
    - `String summary` - итоговая информация
    - `String signature` - подпись
    - `String footnote` - примечание (опционально)
    - `String generationTime` - время генерации (опционально)
- **Конструктор**: принимает итоговую информацию и опционально подпись
- **Метод `render()`**:
    - Возвращает форматированный подвал
    - Может включать разделители, подписи
    - Формат: `--- Итог: ... ---`, `Подпись: ...`

### 3. Класс `ReportBuilder`

Представляет конструктор отчёта (композиция):

- **Поле**: `List<ReportPart> parts` - список частей отчёта
- **Методы**:
    - `void addPart(ReportPart part)` - добавить часть в отчёт
    - `void removePart(ReportPart part)` - удалить часть (опционально)
    - `void addPartAt(int index, ReportPart part)` - добавить часть на позицию (опционально)
    - `String build()` - собрать и вернуть полный отчёт
    - `void clear()` - очистить все части
    - `List<ReportPart> getParts()` - вернуть копию списка частей (опционально)

### 4. Класс `Main`

Демонстрирует работу системы:

- Создаёт различные части отчёта (Header, Table, Footer)
- Создаёт конструктор отчёта
- Добавляет части в разных комбинациях
- Собирает и выводит отчёты
- Показывает, как легко изменить структуру отчёта

## Технические требования

### Композиция

- `ReportBuilder` содержит коллекцию объектов `ReportPart` (has-a отношение)
- Отчёт собирается из независимых компонентов
- Части могут быть добавлены, удалены, переупорядочены

### Полиморфизм через интерфейс

- Все части реализуют интерфейс `ReportPart`
- `ReportBuilder` работает с интерфейсом, а не с конкретными классами

### Расширяемость

- Новые части отчёта (например, `Chart`, `Image`, `Summary`) можно добавить без изменения `ReportBuilder`
- Структура отчёта легко изменяется

### Форматирование

- Каждая часть отвечает за своё форматирование
- Единый стиль отчёта может обеспечиваться через базовые классы

## Ключевые концепции

- **Композиция**: объект состоит из других объектов
- **Интерфейс**: единый контракт для всех частей
- **Полиморфизм**: разные части через единый интерфейс
- **Расширяемость**: добавление новых типов частей
- **Гибкость**: изменение структуры отчёта без изменения кода
