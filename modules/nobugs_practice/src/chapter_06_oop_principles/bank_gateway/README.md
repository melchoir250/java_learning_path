# Задача 21: Платёж через разные банки (Интерфейс + Стратегия)

## Описание проекта

Разработайте систему банковских платежей, демонстрирующую паттерн "Стратегия" через интерфейсы. Платёжный сервис должен
уметь работать с различными банковскими API, не зная деталей их реализации. Каждый банк предоставляет свой способ
обработки платежей, но все они скрыты за единым интерфейсом.

## Структура проекта

```
bank_payments/
├── BankGateway.java          # Интерфейс банковского шлюза
├── SberbankAPI.java          # Реализация для Сбербанка
├── TinkoffAPI.java           # Реализация для Тинькофф
├── AlfaBankAPI.java          # Реализация для Альфа-Банка
├── PaymentService.java       # Платёжный сервис (стратегия)
├── Main.java                 # Демонстрация работы
└── README.md                 # Описание задачи
```

## Функциональные требования

### 1. Интерфейс `BankGateway`

Определяет контракт для всех банковских шлюзов:

- **Метод `boolean transfer(double amount)`** - выполняет перевод указанной суммы
- **Метод `boolean transfer(double amount, String fromAccount, String toAccount)`** - перевод с указанием счетов (
  опционально)
- **Метод `double getBalance(String accountNumber)`** - получение баланса (опционально)
- **Метод `String getBankName()`** - возвращает название банка

### 2. Реализации банковских шлюзов

#### `SberbankAPI`

- Представляет интеграцию с API Сбербанка
- **Поля**:
    - `String apiKey` - ключ API для аутентификации
    - `String baseUrl` - URL API
- **Особенности**:
    - Может требовать специфическую аутентификацию (токен, подпись)
    - Комиссия за перевод (например, 0.5%)
    - Проверка лимитов

#### `TinkoffAPI`

- Представляет интеграцию с API Тинькофф
- **Поля**:
    - `String clientId` - идентификатор клиента
    - `String clientSecret` - секретный ключ
- **Особенности**:
    - Может использовать OAuth2 для авторизации
    - Комиссия за перевод (например, 0.3%)
    - Мгновенные переводы

#### `AlfaBankAPI`

- Представляет интеграцию с API Альфа-Банка
- **Поля**:
    - `String certificate` - сертификат для подписи
    - `String merchantId` - идентификатор мерчанта
- **Особенности**:
    - Требует цифровую подпись для каждой транзакции
    - Комиссия за перевод (например, 0.7%)
    - Возможны ограничения по времени

### 3. Класс `PaymentService`

Представляет платёжный сервис, использующий стратегию:

- **Поле**:
    - `BankGateway gateway` - текущий банковский шлюз (стратегия)
- **Конструктор**: принимает любую реализацию `BankGateway`
- **Методы**:
    - `void setGateway(BankGateway gateway)` - смена банковского шлюза
    - `boolean makePayment(double amount)` - выполнить платёж через текущий шлюз
    - `boolean makePayment(double amount, String from, String to)` - платёж с указанием счетов
    - `void showGatewayInfo()` - вывести информацию о текущем шлюзе

### 4. Класс `Main`

Демонстрирует работу системы:

- Создаёт экземпляры разных банковских шлюзов
- Создаёт платёжный сервис
- Выполняет платежи через разные банки
- Показывает, что сервис не знает, с каким банком работает
- Демонстрирует смену стратегии

## Технические требования

### Паттерн "Стратегия"

- `PaymentService` принимает стратегию (банковский шлюз) через конструктор или сеттер
- Стратегия может быть заменена во время выполнения
- Алгоритм обработки платежа инкапсулирован в конкретных стратегиях

### Полиморфизм через интерфейс

- Все банковские шлюзы реализуют единый интерфейс
- `PaymentService` работает с интерфейсом, а не с конкретными реализациями
- Добавление нового банка не требует изменения `PaymentService`

### Инкапсуляция

- Детали API каждого банка скрыты внутри реализации
- Платёжный сервис не знает о комиссиях, аутентификации и специфике банков

### Обработка ошибок

- Выброс исключения `PaymentException` при ошибке платежа
- Обработка различных кодов ошибок от банковских API
- Логирование неудачных транзакций

## Ключевые концепции

- **Стратегия (Strategy)**: выбор алгоритма во время выполнения
- **Полиморфизм**: единый интерфейс для разных реализаций
- **Инкапсуляция**: сокрытие сложности банковских API
- **Слабая связанность**: платёжный сервис не зависит от конкретных банков

## Дополнительные требования (опционально)

1. **Комиссии**:
    - Добавить метод `double calculateFee(double amount)` в интерфейс
    - Каждый банк возвращает свою комиссию
    - Платёжный сервис может отображать итоговую сумму с комиссией

2. **История платежей**:
    - Сохранение всех транзакций
    - Метод для получения истории

3. **Валюты**:
    - Поддержка разных валют
    - Конвертация по курсу банка

4. **Лимиты**:
    - Проверка лимитов на перевод в каждом банке
    - Выброс исключения при превышении лимита

5. **Аутентификация**:
    - Метод `boolean authenticate()` для проверки валидности учётных данных
    - Автоматическое обновление токенов

6. **Ретри (повторные попытки)**:
    - Автоматический повтор при временных ошибках
    - Настройка количества попыток

## Критерии оценки реализации

1. **Интерфейс**:
    - Чётко определённый контракт `BankGateway`
    - Все реализации корректно следуют контракту

2. **Паттерн "Стратегия"**:
    - `PaymentService` принимает стратегию извне
    - Возможность смены стратегии во время выполнения

3. **Полиморфизм**:
    - Один и тот же код `PaymentService` работает с любым банком
    - Добавление нового банка не требует изменений в сервисе

4. **Инкапсуляция**:
    - Детали API каждого банка скрыты
    - Платёжный сервис не знает о внутренней логике банков

## Как должен работать код

### Этап 1: Работа с разными банками

1. Создаются объекты `SberbankAPI`, `TinkoffAPI`, `AlfaBankAPI`
2. Создаётся `PaymentService` с одним из банков
3. Вызывается `makePayment()` с суммой
4. Платёжный сервис делегирует вызов конкретному банку
5. Банк выполняет свою логику (аутентификация, комиссия, проверка лимитов)

### Этап 2: Смена стратегии

1. Платёжный сервис меняет банковский шлюз через `setGateway()`
2. Тот же метод `makePayment()` теперь использует новый банк
3. Код, вызывающий платёж, не меняется

### Этап 3: Расширение

1. Создаётся новый класс `VTBAPI`, реализующий `BankGateway`
2. Экземпляр передаётся в `PaymentService`
3. Система работает без изменений
