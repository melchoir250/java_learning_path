## Задача 1: Универсальный менеджер сущностей

### Описание
Разработайте класс `EntityManager<T>`, который будет управлять коллекцией объектов произвольного типа `T`, обеспечивая потокобезопасное добавление, удаление и получение элементов. Класс также должен предоставлять специфические методы фильтрации данных, которые позволяют пользователю извлекать элементы по определённым критериям.

### Функциональные требования

1. **Добавление элементов**: Метод для добавления объекта в коллекцию. Должен быть потокобезопасным.
2. **Удаление элементов**: Метод для удаления объекта из коллекции. Возвращает `true`, если объект был удалён, и `false`, если объект не найден. Должен быть потокобезопасным.
3. **Получение всех элементов**: Метод возвращает **копию** списка всех элементов, обеспечивая невозможность изменения исходной коллекции через возвращаемый список.
4. **Специализированные методы фильтрации** (для демонстрации можно использовать конкретный тип, например, `User`):
    - Фильтрация по возрасту: возвращает список пользователей в заданном возрастном диапазоне.
    - Фильтрация по имени: возвращает список пользователей, чьи имена соответствуют заданной строке.
    - Фильтрация по активности: возвращает список пользователей с заданным статусом активности.

### Ключевые темы
- Дженерики (обобщённый тип `T`)
- Многопоточность и потокобезопасность (`synchronized` или потокобезопасные коллекции)
- Копирование коллекций для защиты от изменений
- Фильтрация с использованием Stream API или циклов

## Задача 2: Валидатор пользовательских данных

### Описание
Разработайте систему валидации для модели данных пользователя, которая проверяет корректность имени, возраста и электронной почты. Валидация должна управляться через глобальный флаг `validationEnabled`, который может быть включен или выключен. Если данные не проходят валидацию, должно выбрасываться специализированное исключение `InvalidUserException`.

### Модель данных
- **`User`**: Класс пользователя с атрибутами для имени, возраста и электронной почты.

### Класс валидатора
- **`UserValidator`**: Сервис, который предоставляет методы для проверки объектов `User` на соответствие определённым правилам.

### Функциональные требования

1. **Проверка имени**: Имя должно быть не пустым и начинаться с заглавной буквы.
2. **Проверка возраста**: Возраст должен быть в пределах от 18 до 100 лет.
3. **Проверка email**: Email должен соответствовать стандартному формату электронной почты (можно использовать регулярное выражение).
4. **Управление валидацией**: Валидация данных должна происходить только если флаг `validationEnabled` установлен в `true`.
5. **Исключения**: При обнаружении невалидных данных необходимо выбрасывать `InvalidUserException` с информативным сообщением.

### Ключевые темы
- Классы и объекты
- Регулярные выражения для валидации email
- Пользовательские исключения
- Условная логика (флаг включения/выключения)

## Задача 3: Учёт оценок студентов

### Описание
Создайте систему для управления и анализа оценок студентов, используя обобщённый подход. Система должна поддерживать различные типы числовых оценок, обеспечивать валидацию входных данных и предоставлять функционал для расчёта статистических показателей.

### Функциональные требования

1. **Класс `StudentGrade<T>`**:
    - Поля для имени студента, предмета и оценки.
    - Оценка должна быть типа `T`, который расширяет класс `Number` (`T extends Number`).
    - Конструктор для инициализации всех полей.
    - Геттеры для доступа к полям.

2. **Сервис `GradeService<T>`**:
    - Список `List<StudentGrade<T>>` для хранения оценок.
    - Метод для добавления оценки (`addGrade`), который также валидирует оценку на предмет того, что она не отрицательна.
    - Метод для расчёта среднего значения оценок по конкретному предмету.
    - Обработка исключений через `InvalidGradeException`, если оценка некорректна.

3. **Многопоточность**:
    - Обеспечение потокобезопасности при добавлении оценок с использованием `synchronized`.

### Ключевые темы
- Дженерики с ограничениями (`<T extends Number>`)
- Валидация данных и пользовательские исключения
- Stream API для расчёта среднего
- Синхронизация потоков

## Задача 4: Рейтинг фильмов

### Описание
Разработайте систему, которая позволяет пользователям ставить оценки фильмам и управлять этими оценками. Система должна поддерживать использование различных типов рейтингов, таких как целые числа или дробные значения, и предоставлять функциональность для расчёта средней оценки фильма, а также для сортировки фильмов по популярности.

### Функциональные требования

1. **Классы и интерфейсы**:
    - **`Movie`**: Класс, представляющий фильм с полями для названия и других характеристик.
    - **`Rating<T extends Number>`**: Класс для хранения рейтинга фильма. `T` может быть `Integer`, `Double` и т.д.

2. **Сервис `MovieService`**:
    - Хранение оценок в `Map<Movie, List<Rating<?>>>`.
    - Метод для добавления оценки к фильму. Метод должен быть потокобезопасным и валидировать оценку на допустимость (например, от 1 до 10).
    - Возможность расчёта средней оценки для каждого фильма.

3. **Обработка данных**:
    - Использование Stream API для подсчёта средней оценки.
    - Использование Stream API и лямбда-выражений для сортировки фильмов по средней оценке.

### Ключевые темы
- Дженерики в коллекциях
- Вложенные структуры данных (`Map<Movie, List<Rating>>`)
- Stream API: `mapToDouble()`, `average()`, `sorted()`
- Лямбда-выражения для компараторов
- Потокобезопасность при добавлении

## Задача 5: Инвентарь склада

### Описание
Разработайте систему для управления складом, которая позволяет добавлять и извлекать товары различных типов. Система должна управлять доступом к складу через переменную `isInventoryOpen` и обрабатывать ситуации, когда товары заканчиваются, выбрасывая исключение `OutOfStockException`.

### Функциональные требования

1. **Классы и интерфейсы**:
    - **`Product`**: Продукт с минимальными характеристиками: название, цена, категория.
    - **`InventoryService`**: Сервис для управления товарами на складе.

2. **Управление товарными запасами**:
    - Хранение товаров в структуре `Map<String, List<Product>>`, где ключ — категория товара.
    - Метод для добавления товара на склад. Если флаг `isInventoryOpen` равен `false`, операция добавления не должна выполняться.
    - Метод для получения товара по категории. Если в указанной категории нет товаров, должно выбрасываться исключение `OutOfStockException`.

3. **Работа с данными**:
    - Использование Stream API для поиска и фильтрации товаров по категориям.
    - Фильтрация товаров по цене с использованием лямбда-выражений.

### Ключевые темы
- Коллекции: `Map<String, List<Product>>`
- Управление состоянием (флаг `isInventoryOpen`)
- Пользовательские исключения
- Stream API и лямбды для фильтрации

## Задача 6: Сервис задач (Task Manager)

### Описание
Разработайте сервис для управления задачами, который позволяет добавлять, удалять и искать задачи по различным критериям. Каждая задача будет иметь уникальный идентификатор обобщённого типа, статус, приоритет и дату.

### Функциональные требования

1. **Классы и интерфейсы**:
    - **`Task<T>`**: Класс, представляющий задачу. Содержит поля для ID (типа `T`), статуса, приоритета и даты.
    - **`TaskService<T>`**: Сервис для управления задачами.

2. **Управление задачами**:
    - **Добавление задачи**: Метод для добавления новой задачи. Желательно проверять уникальность ID.
    - **Удаление задачи**: Метод для удаления задачи по ID. Должен быть синхронизирован для предотвращения конкурентного доступа.
    - **Поиск задач**: Методы для фильтрации задач по статусу и приоритету, а также сортировка задач по дате.

3. **Обработка данных**:
    - Использование Stream API для фильтрации и сортировки задач.
    - Лямбда-выражения для сортировки задач по дате.

### Ключевые темы
- Дженерики для идентификатора
- Синхронизация при удалении
- Stream API: `filter()`, `sorted()`, `collect()`
- Лямбда-выражения для компараторов
- Работа с датами
